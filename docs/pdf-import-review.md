## 01 · What we built

In the order it happened — each piece builds on the last.

1. **PDF → text extraction engine** — *committed `b7253ad`*
   New `pdf-text-extract.mjs` reads text out of your PDFs using the **PDF.js that
   Foundry already bundles**. Extraction runs locally — nothing is sent to an
   external service, and the module bundles no book content. (The PDFs themselves
   live in your Foundry data and are served to the browser, as with any local
   asset.) Column-aware: it finds the gutter on two-column pages with a
   density-valley detector and reads each column in order.

2. **"Horizontal Pages" rotation fix** — *committed `b7253ad`*
   Some Cursed Scrolls ship a re-saved printing that rotates map/spread pages
   90°. Pinning `rotation: 0` when reading page width makes both printings
   extract **byte-identical**, so either can be registered.

3. **Grab text · Extract from PDF · review flags** — *committed `b7253ad`*
   Importer Hub gained one-click **Grab text** (seed → cited page) and a
   **page-range Extract** dialog. Monster cards now flag the exact attack/feature
   row a warning names, with the reason on hover.

4. **Monster-census extraction path** — *committed `ed5fae5`*
   Monster bundle rows carry no page cite, so they got source-aware seeding and
   an inline **Grab from PDF** that pre-selects the right book.

5. **Table parser hardening** — *uncommitted · now tested*
   Two distinct fixes (see §07 for the page-number distinction):
   - `repairSharedStartRanges()` — a **general heuristic**: when a row's low bound
     repeats the previous row's low bound, it shifts to `prev.max + 1` with an
     `Auto-fixed:` note. It was prompted by the WR Dwarf Trinket table, whose PDF
     literally prints `21-24` where it means `23-24`, but it applies to any table.
   - An **empty-row filter** that drops a bare number left on its own line (a
     page number swept in by extraction) before it becomes a phantom row.
   Both now have focused positive/negative tests in
   `test/parser-review-regressions.test.mjs`.

6. **Import auto-extracts & auto-parses** — *uncommitted*
   Pressing **Import** now pulls the cited page in and parses it in one action.
   Two bugs found and fixed along the way: a render-ordering bug (stale textarea
   read) and the stray page-number row (§07 bug #1).

7. **Class importer: full content + reviewable preview** — *uncommitted*
   Class Import pulls the writeup, the talent table, **and** the titles (a
   separate appendix page, read single-column and sliced per class). The preview
   makes the talent table, titles, extra tables (e.g. Wyrdling Corruption) and
   the spells-known grid all reviewable/editable before submitting, with title
   errors flagged in place.

8. **Manage coverage audit** — *sweep artifact + probe retained in `docs/`*
   Swept the 184 page-cited entries through the real import path (the 10 spell
   lists log as workspace-routed), registered all 8 source PDFs, and triaged
   every table that still needs review. This document is that audit; the raw
   per-entry data is retained.

## 04 · Key technical findings

- **Foundry ships PDF.js.** The whole extractor rides on the library Foundry
  already bundles. No external service; the module bundles no book content.
- **Your books have real typos.** The Dwarf Trinket `21-24` is in the printed
  PDF (spot-checked against an independent text extraction of the page). The
  extractor is faithful — the parser now auto-repairs the pattern with a visible
  note, and the repair is a general heuristic, not table-specific.
- **Titles are graphics, not text.** Stylized class titles ("DELVER") aren't in
  the text layer at all. The class name is supplied by the unlock.
- **Numeric columns mangle.** Narrow roll-number / level columns (corruption
  d10, spells-known grids, some name tables) separate from their text on
  extraction. Single-column reading rescues titles but not these.
- **The shape system is doing the work.** Prayers, carousing (Event), traps, and
  generators only look broken under the generic parser; their deterministic shape
  recipes handle them cleanly. *Carousing Outcome* is the one shaped table still
  warning.
- **One extractor change already backfired.** A horizontal-banding attempt
  regressed clean pages and was reverted. Extractor changes are
  high-blast-radius — verify against the full sweep.

## 05 · Parsers & the router question

On "too many parsers," and whether a router should direct traffic.

There are two very different things called "parsers" here, and only one carries
real risk from growth:

- **Auto-detect recognizers** (the segmenter cascade) run on freeform pastes and
  compete to claim text. More of these = more mis-classification. This is where
  "too many" genuinely bites.
- **Shape recipes** are name-matched — they only fire on a matching seed, so they
  don't compete on arbitrary input. Adding here is far safer.

`shapeForName()` today matches **exact normalized names plus delimiter-aware
prefixes** (`… - <shape>` / `… : <shape>`) — not general fuzzy matching. A future
*generic* shape name could still collide with an unrelated prefixed name, which
is the real argument for stable IDs.

**A router is largely the right model — and much of it already exists.** Manage
entries carry `{ name, type, src, pages }`
([`manage-tree.mjs`](../scripts/encounter/manage-tree.mjs) lines 10–14, 72–80).
The importer already routes explicitly: seed types map to importer modes, classes
and spell lists open dedicated workspaces, shape lookup runs before generic
parsing, and the competing segmenter runs only when the mode is `auto`
([`importer-hub-app.mjs`](../scripts/encounter/importer-hub-app.mjs) ~2412–2475).

So the work is **registry consolidation**, not a brand-new router:

- Give each known manifest entry a **persistent `contentId`** — a display name
  plus page citation is *not* stable (names get corrected, pages shift between
  printings, the same name recurs across sources).
- Resolve known seeded content by exact `contentId` lookup; keep the type-specific
  routes and the `auto`-only segmenter fallback.
- Fold `TABLE_SHAPES` into the registry keyed by `contentId`, retiring name-based
  resolution and its collision risk.
- Routing does **not** fix extraction mangling — the two are complementary.

## 06 · Known limitations

Honest edges — none of them block import; all have a hand path.

- **Numeric-column tables don't auto-parse** (corruption d10, spells-known grids,
  some name tables). Scaffolded: the text is pulled in and the preview is
  editable, so you build/fix by hand.
- **Two-word titles can't single-space-split** (e.g. "Tomb Robber") — flagged for
  review and editable in place.
- **Necromancer & caster spells-known** don't parse yet — the grid interleaves in
  the two-column writeup.
- **Monster bundle rows have no page cite** — they use the manual page-range
  Extract rather than one-click.
- **CSS changes need a hard refresh** (Ctrl+Shift+R) — Foundry caches module CSS
  across reloads.

## 07 · Two page-number bugs, kept distinct

The review correctly flagged that "page numbers" show up as *two* separate
failures. They are:

1. **Bare page-number row.** A page number left alone on a line parses to a range
   row with empty text. **Fixed** by the empty-row filter in `parseSingleDieBlock`
   (§01.5), with a regression test.
2. **Page/stray number in the formula or range inference.** In a multi-column
   table without a shape, a page/large number contaminates the max-range → a
   formula like `1d284` (p284), `1d290` (p290), `1d134` (p134). **Not yet fixed** —
   this is Recommendation #1.

Both surface in the sweep JSON via the `formula` field.

## 09 · Recommendations going forward

Sequenced so each step is independently verifiable and lands on solid ground.

1. **Fix the page-number formula pollution, then re-sweep.** *(quickest win)*
   Stop a page/stray number from being read into the max-range (bug #2 in §07).
   Re-run the probe and diff the JSON against the retained baseline to see how
   many of the 38 it clears before deciding which recipes are actually needed.

2. **Consolidate dispatch into a registry.** *(architecture)*
   A `resolveHandler(entry)` keyed on a persistent `contentId`: known content is
   dispatched by exact lookup, `TABLE_SHAPES` moves under it, and name-based shape
   resolution (with its collision risk) goes away. This mostly formalizes routing
   that already exists.

3. **Add the remaining generator shapes as registry configs.** *(coverage)*
   Only for genuine multi-column generators, keyed on `contentId`, with a live
   check each. The retained sweep is the acceptance test.

4. **Tackle Bucket B/C extraction & caster spells-known.** *(hard cases)*
   The multi-column long tables (WR Backgrounds) and the caster spells-known grid
   need real extraction work. Scaffolds keep them usable in the meantime.

5. **Add focused tests, then commit the open work in logical batches.**
   *(housekeeping)* Before committing, add regression tests for the still-untested
   paths listed in §08. Suggested batches: table-parser fixes · auto-extract-on-
   Import · class-import + preview. Keep the pre-existing terminology rewrite and
   `gear-join.mjs` out, as we have been.
