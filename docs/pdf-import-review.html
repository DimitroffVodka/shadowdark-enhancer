What we built

In the order it happened — each piece builds on the last.

01

PDF → text extraction engine committed b7253ad

New pdf-text-extract.mjs reads text out of your PDFs using the PDF.js that Foundry already bundles. Extraction runs locally — nothing is sent to an external service, and the module bundles no book content. Column‑aware: it finds the gutter on two‑column pages with a density‑valley detector and reads each column in order.

02

“Horizontal Pages” rotation fix committed b7253ad

Some Cursed Scrolls ship a re‑saved printing that rotates map/spread pages 90°. Pinning rotation: 0 when reading page width makes both printings extract byte‑identical.

03

Grab text · Extract from PDF · review flags committed b7253ad

Importer Hub gained one‑click Grab text and a page‑range Extract dialog. Monster cards flag the exact attack/feature row a warning names, with the reason on hover.

04

Monster‑census extraction path committed ed5fae5

Monster bundle rows carry no page cite, so they got source‑aware seeding and an inline Grab from PDF that pre‑selects the right book.

05

Table parser hardening uncommitted now tested

Two distinct fixes (see §07). A general shared‑start range repair — prompted by the WR Dwarf Trinket table, whose PDF prints 21‑24 where it means 23‑24, but it applies to any table. And an empty‑row filter that drops a bare page number before it becomes a phantom row. Both now have regression tests.

06

Import auto‑extracts & auto‑parses uncommitted

Pressing Import now pulls the cited page in and parses it in one action. Two bugs found and fixed along the way: a render‑ordering bug (stale textarea read) and the stray page‑number row (§07 bug #1).

The preview makes the talent table, titles, extra tables (e.g. Wyrdling Corruption) and the spells‑known grid reviewable/editable before submitting, with title errors flagged in place.

04

Key technical findings

The things that shaped every decision.

Foundry ships PDF.js

The whole extractor rides on the bundled library. No external service; the module bundles no book content.

Your books have real typos

The Dwarf Trinket 21‑24 is in the printed PDF (spot‑checked independently). The repair is a general heuristic, not table‑specific — now covered by tests.

Titles are graphics, not text

Stylized class titles (“DELVER”) aren’t in the text layer. The class name is supplied by the unlock.

Numeric columns mangle

Narrow roll‑number / level columns (corruption d10, spells‑known grids, some name tables) separate from their text. Single‑column rescues titles but not these.

The shape system carries most tables

Prayers, carousing Event, traps, and generators only look broken under the generic parser. Carousing Outcome is the one shaped table still warning.

One extractor change backfired

A horizontal‑banding attempt regressed clean pages and was reverted. Extractor changes are high‑blast‑radius — verify against the full sweep.

05

Parsers & the router question

On “too many parsers,” and whether a router should direct traffic.

Two very different things are called “parsers,” and only one carries real risk from growth:

  • Auto‑detect recognizers (the segmenter cascade) run on freeform pastes and compete to claim text. More of these = more mis‑classification. This is where “too many” bites.
  • Shape recipes are name‑matched — they only fire on a matching seed, so they don’t compete on arbitrary input.

shapeForName() matches exact normalized names plus delimiter‑aware prefixes (… - name / … : name) — not general fuzzy matching. A future generic shape name could still collide with an unrelated prefixed name, which is the real argument for stable IDs.

A router is largely the right model — and much of it already exists. Manage entries carry { name, type, src, pages }, and the importer already routes explicitly: seed types map to modes, classes and spell lists open dedicated workspaces, shape lookup runs before generic parse, and the segmenter runs only in auto mode. So the work is registry consolidation: give each manifest entry a persistent contentId (a display name + page is not stable), resolve known content by exact lookup, and fold TABLE_SHAPES under it to retire name‑based resolution. Routing does not fix extraction mangling — the two are complementary.
06

Known limitations

Honest edges — none block import; all have a hand path.

  • Numeric‑column tables don’t auto‑parse (corruption d10, spells‑known grids, some name tables). Scaffolded: text pulled in, preview editable.
  • Two‑word titles can’t single‑space‑split (e.g. “Tomb Robber”) — flagged and editable in place.
  • Necromancer & caster spells‑known don’t parse yet — the grid interleaves in the two‑column writeup.
  • Monster bundle rows have no page cite — they use manual page‑range Extract.
  • CSS changes need a hard refresh (Ctrl+Shift+R) — Foundry caches module CSS.
07

Two page‑number bugs, kept distinct

“Page numbers” cause two separate failures — one fixed, one not.

Bare page‑number row — fixed

A page number alone on a line parses to an empty‑text range row. Dropped by the empty‑row filter in parseSingleDieBlock, with a regression test.

Number in formula/range — open

In an un‑shaped multi‑column table, a page/large number contaminates the max‑range → a formula like 1d284. This is Recommendation #1.

Recommendations going forward

Sequenced so each step is independently verifiable.

1
quickest win

Fix the page‑number formula pollution, then re‑sweep

Stop a page/stray number reaching the max‑range (bug #2, §07). Re‑run the probe and diff the JSON against the retained baseline before deciding which recipes are needed.

2
architecture

Consolidate dispatch into a registry

A resolveHandler(entry) keyed on a persistent contentId: exact lookup for known content, TABLE_SHAPES folded under it, name‑based resolution retired. Mostly formalizes routing that already exists.

3
coverage

Add the remaining generator shapes as registry configs

Only for genuine multi‑column generators, keyed on contentId, with a live check each. The retained sweep is the acceptance test.

4
hard cases

Tackle Bucket B/C extraction & caster spells‑known

The multi‑column long tables (WR Backgrounds) and the caster spells‑known grid need real extraction work. Scaffolds keep them usable meanwhile.

5
housekeeping

Add focused tests, then commit in logical batches

Cover the still‑untested paths from §08 first. Suggested batches: table‑parser fixes · auto‑extract‑on‑Import · class‑import + preview. Keep the pre‑existing terminology rewrite and gear‑join.mjs out.
